# Backend Implementation Notes

Purpose: capture the verified backend state from code inspection so future backend work starts from source-backed facts.

Backend root: `../../Backend`.

## Runtime Composition

Entry point: `AssetManagement.API/Program.cs`.

Startup order:

1. Registers infrastructure with `AddInfrastructure(builder.Configuration)`.
2. Registers application services with `AddApplication()`.
3. Registers API services with `AddApiServices(builder.Configuration)`.
4. Binds `DataRetentionOptions` from `DataRetention`.
5. Registers `DataCleanupBackgroundService`.
6. Adds authorization.
7. Enables Swagger only in development.
8. Adds middleware: exception handling, CORS `AllowAll`, HTTPS redirection, authentication, authorization, audit logging.
9. Maps controllers.
10. Creates a scope and calls `DataSeeder.SeedAsync`.

Important: the code comment says "Database Migration & Seeding" and "Apply migrations", but the inspected code does not call `Database.MigrateAsync()`. It only seeds when the user table is empty.

## Dependency Registration

`AssetManagement.Infrastructure/DependencyInjection.cs` registers:

- EF Core `ApplicationDbContext` with PostgreSQL and migrations assembly set to infrastructure.
- Repository implementations for asset, assignment, user, location, category, audit log, and unit of work.
- `PasswordHasher` and `JwtTokenGenerator`.
- JWT bearer authentication using `JwtSettings`.

`AssetManagement.Application/DependencyInjection.cs` registers:

- AutoMapper with `MappingProfile`.
- FluentValidation validators from the application assembly.
- Application services for assets, assignments, users, locations, categories, auth, audit, and data cleanup.

`AssetManagement.API/Extensions/ServiceExtensions.cs` registers:

- Controllers with `ValidationFilter`.
- CORS policy named `AllowAll`.
- Swagger/OpenAPI with bearer auth definition.

## Configuration Reality

Current backend config files contain operational values:

- `ConnectionStrings:DefaultConnection` points to local PostgreSQL database `AssetManagementDB` with `postgres/postgres`.
- `JwtSettings:Secret` is committed and used for signing tokens.
- `JwtSettings:Issuer` is `AssetManagement.API`.
- `JwtSettings:Audience` is `AssetManagement.Client`.
- `JwtSettings:ExpirationInMinutes` is `60`.
- `DataRetention` exists in `appsettings.json`; development config does not override it.

Treat these as local defaults only. See `deployment-config-notes.md` before production or shared-environment work.

## Authentication and Authorization

Authentication:

- Login uses email/password in `AuthService.LoginAsync`.
- Passwords are verified through `IPasswordHasher`.
- JWT is generated by `IJwtTokenGenerator`.
- Logout is client-side token removal; backend `LogoutAsync` returns success and does not revoke tokens.
- Current user lookup reads `ClaimTypes.NameIdentifier`.

Authorization:

- Controllers mostly use `[Authorize]` at class level.
- Write operations generally restrict to `Admin` or `Admin,Manager`.
- `DataRetentionController` is Admin-only.
- `AuditLogsController` is Admin-only.

Backend authorization is the source of truth. Frontend guards and role visibility must mirror it, not replace it.

## API Response Pattern

Controllers mostly call an application service returning `Result<T>` or `Result`.

Common pattern:

- success returns `Ok(...)`
- service failure returns `BadRequest(result.Error)` or `NotFound(result.Error)`
- auth failure returns `Unauthorized(...)`

Exception middleware exists, but normal business failures generally flow through `Result` rather than exceptions.

## Validation

`ValidationFilter` is registered globally on MVC controllers. FluentValidation validators exist for create/login DTOs and are registered from the application assembly.

When adding request DTOs, add validators for boundary rules and keep deeper business rules in services.

## Persistence and Soft Delete

The infrastructure layer owns EF Core configuration, repositories, query filters, migrations, and cleanup queries.

Soft delete and restore are explicit concepts:

- Category, location, user, and asset controllers expose restore endpoints.
- Repositories include methods for deleted records and cleanup.
- Data cleanup permanently removes soft-deleted records after retention cutoff or immediately during manual cleanup.

Before changing delete behavior, inspect entity, repository, query filters, restore service method, cleanup service, and frontend list reloads.

## Data Seeding

`DataSeeder.SeedAsync` exits if any user already exists. Otherwise it creates:

- default categories
- default locations
- admin user `admin@assetmanagement.local` with password `Admin123!`
- sample assets

This is useful for local development but unsafe as an unconditional production behavior. Make seeding environment-aware or externally configured before deployment.

## Audit Logging

`AuditLoggingMiddleware` runs after the request pipeline and logs successful `POST`, `PUT`, and `DELETE` requests for authenticated users.

Current limitations:

- Entity id is always `Guid.Empty`.
- Entity name is derived from the URL path segment.
- Changes are not captured.
- Route-specific business context is not captured.
- It logs after controller execution, but outside the specific use-case transaction.

For precise audit requirements, prefer explicit service-level audit calls or enrich middleware with route value extraction and response/request context.

## Data Retention and Cleanup

`DataCleanupBackgroundService` runs continuously:

- Executes cleanup immediately on service loop start.
- Waits 24 hours after successful cleanup.
- Waits 1 hour before retry after errors.
- The 24-hour interval is hardcoded, not configurable.

`DataCleanupService.PerformCleanupAsync(ignoreRetention: false)` uses configured cutoffs:

- categories: hours
- locations: hours
- users: days
- assets: days
- audit logs: hours

`DataRetentionController.ExecuteManualCleanup()` calls `PerformCleanupAsync(ignoreRetention: true)`, which permanently deletes all currently soft-deleted records and all audit logs selected by cleanup-all repository methods. Treat this endpoint as destructive Admin-only maintenance.

## Known Backend Risks

- Startup does not apply migrations despite comments suggesting it does.
- CORS is open to all origins/methods/headers.
- JWT signing secret is committed.
- PostgreSQL credentials are committed.
- Seeded admin credentials are deterministic.
- JWT logout does not invalidate issued tokens.
- Audit middleware records weak audit data.
- Manual cleanup ignores retention.
- Cleanup interval is hardcoded.
- `DataRetentionOptions` contains notification settings, but warning notification logic is placeholder-level.
- There are no backend test projects in the inspected file list.

## Recommended Backend Next Steps

1. Move secrets and connection strings to user-secrets/environment variables.
2. Make seeding environment-aware and disable deterministic admin creation outside development.
3. Decide whether startup should run migrations; if yes, add explicit `Database.MigrateAsync()` with operational safeguards.
4. Replace `AllowAll` CORS with environment-configured allowed origins.
5. Improve audit logging entity id/change capture.
6. Make cleanup interval configurable and add safer confirmation/preview for manual cleanup.
7. Add backend test projects for services and controller/auth integration behavior.
